# EnigmaHack_Quadro
решение для хакатона enigma hack: ии-агент классификации писем, извлечения данных и генерации ответов на основе базы данных


### Пользовательский путь (User Journey)

1.  **Поступление письма**
    - Пользователь (клиент) отправляет письмо на адрес технической поддержки компании (например, support@company.ru).
    - Система через настроенный почтовый ящик (Яндекс, Mail.ru или Gmail) по протоколу IMAP периодически проверяет новые непрочитанные сообщения.
    
2. **Автоматическая обработка ИИ-агентом**
    - Бэкенд получает текст письма и метаданные (отправитель, дата, тема).
    - Запускается ML-модуль, который выполняет:
        - Извлечение ключевых данных (NER): 
	        - ФИО, 
	        - название объекта, 
	        - телефон, 
	        - email, 
	        - заводские номера приборов,
	        - тип прибора,
	        - суть вопроса.
        - Анализ тональности: определение эмоционального окраса (позитив/нейтраль/негатив) с помощью предобученной NLP-модели.
        - Классификация запроса: отнесение письма к одной из категорий (например, "неисправность", "калибровка", "запрос документации" и другие) на основе модели классификации.
    - Извлечённые данные структурируются и сохраняются в базе данных (таблица `messages`).
    
3. **Генерация проекта ответа**
    - На основе определённой категории система обращается к базе знаний (таблица `knowledge_base`), где хранятся шаблоны ответов, FAQ, выдержки из руководств по эксплуатации.
    - В выбранный шаблон подставляются извлечённые сущности (например, номер прибора, имя клиента).
    - Сгенерированный ответ сохраняется в БД как черновик.
    
4. **Заполнение веб-таблицы**
    - Фронтенд (веб-интерфейс) периодически запрашивает через REST API список обращений.
    - Таблица автоматически обновляется, отображая все поля, требуемые в кейсе:
	    - дата поступления
	    - ФИО отправителя
	    - объект (предприятие)
	    - телефон
	    - email
	    - заводские номера приборов
	    - тип приборов
	    - эмоциональный окрас
	    - суть вопроса
	    - генерированный ответ
	    - статус (новое/обработано/отправлено)
    
5. **Работа оператора с обращением**
    - Оператор видит новые обращения, может:
        - Просмотреть детали письма.
        - Отредактировать сгенерированный ответ при необходимости.
        - Отфильтровать и отсортировать записи по любому полю.
        - Экспортировать таблицу в CSV/XLSX для отчётности.        
    - Если всё устраивает, оператор нажимает кнопку «Отправить ответ».    
    
6. **Отправка ответа клиенту**
    - Backend получает команду на отправку, использует SMTP-соединение (с того же почтового ящика) для отправки письма клиенту.
    - Статус обращения в БД меняется на «отправлено».
    
7. **Уведомления (опционально)**
    - При поступлении нового письма может отправляться уведомление в Telegram/Slack операторам для ускорения реакции.

### Компоненты

1. **Backend (FastAPI)**  
	- REST API для взаимодействия с фронтендом.
	- Модуль получения почты (IMAP) периодически проверяет новые письма.
	- Модуль отправки почты (SMTP) отправляет ответы.
	- Интеграция с ML-модулем и базой знаний.
	- Хранение данных в PostgreSQL.
	- Эндпоинты для экспорта данных в CSV/XLSX.
	
2. **ML-модуль**  
	- Извлечение сущностей (NER): комбинация правил (регулярные выражения для номеров, телефонов) и библиотеки Natasha (для имён, дат, адресов).
	- Анализ тональности: предобученная модель из Hugging Face.
	- Классификация запросов: модель на базе `rubert-tiny2`, дообученная для определения категории.
	- Возможность замены на rule-based заглушку для ускорения разработки.
3. **База данных (PostgreSQL)** 
	- `messages` — все обращения с полями согласно структуре веб-таблицы.
	- `knowledge_base` — связка «категория - шаблон ответа», а также дополнительные материалы (FAQ, руководства).
	- `extracted_entities` — для хранения извлечённых данных в нормализованном виде (опционально).
4. **Frontend (HTML, CSS, JavaScript)**
	- Таблица обращений с фильтрацией и сортировкой (реализована на чистом JS).
	  - Форма для ручного ввода/тестирования письма.
	  - Детальный просмотр письма с редактированием ответа.
	  - Кнопка отправки ответа.
	  - Кнопка экспорта таблицы в CSV/XLSX (с использованием библиотеки типа `xlsx` или генерации CSV на лету).
	  - Адаптивный и минималистичный дизайн.
5. **Docker** 
	- Контейнеризация всех сервисов для лёгкого развёртывания (в процессе).


### Риски и их минимизация
| Риск                                                                   | Минимизация                                                                                                                                                                                             |
| ---------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Сложность интеграции IMAP/SMTP с конкретным почтовым сервисом          | Используем Яндекс.Почту как основной, у неё простая настройка, есть готовые примеры кода. Код написан универсально, смена провайдера требует только изменения переменных окружения.                     |
| Нехватка времени на дообучение ML-моделей (тональность, классификация) | Готовим rule-based заглушки, которые смогут демонстрировать логику работы. Используем предобученные модели zero-shot, где это возможно. На хакатоне при наличии данных быстро дообучаем или адаптируем. |
| Извлечение сущностей в свободном формате писем                         | Комбинируем Natasha (для имён, дат) и регулярные выражения (для номеров, телефонов). При необходимости дополняем правилами на основе анализа предоставленных организаторами примеров.                   |
| Конфликты при работе в одной ветке                                     | Чёткое разделение по папкам (backend, frontend, ml, database). Перед началом работы делаем `git pull`, коммиты подписываем с указанием автора.                                                          |
| Отсутствие опыта у некоторых участниц                                  | Используем понятные технологии (FastAPI, нативный JavaScript, готовые библиотеки). Планируем парное программирование и взаимопомощь.                                                                    |
| Проблемы с доступом к почтовому ящику во время демо                    | Создадим резервный тестовый ящик и подготовим моковые данные для демонстрации без реальной почты.                                                                                                       |
| Сложность реализации экспорта в XLSX                                   | Используем лёгкую библиотеку `xlsx` на фронтенде или формируем CSV (поддерживается Excel).                                                                                                              |
